## The work — wire the 5 un-driven models + fix auto_fill

### 1. EventSubscription (highest value — 7 fields currently un-checked)
The event-source seeders (`seed_event_source`, `seed_solution_event_source`) create sources with NO subscriptions, so `ManifestEventSubscription`'s fields (`target_type`, `workflow_id`, `agent_id`, `event_type`, `filter_expression`, `input_mapping`, `is_active`) never travel. Extend the seeders to attach a subscription (a workflow dep is already available via the workflow seeder pattern), so the subscription is serialized and the harness asserts its fields. It's nested under EventSource — assert it the way nested children are paired (within the matched parent; `workflow_id`/`agent_id` are references → exact-id remap on the solution path).

### 2. Organization + Role (easy — simple field sets)
Both currently travel only as closure under the claim seeder, not field-asserted. Drive them field-by-field (`_repo` is enough — they're identity entities, not in `SolutionBundle`). `name` is the match key; both are small.

### 3. Policy (medium)
`ManifestPolicy` lives inside `Table.access` as an opaque blob — asserted today only as part of the table CONTENT blob, not field-by-field. Crack it: seed a table with a real policy (name/description/actions/when), and assert the policy's fields survive the round trip individually (not just byte-equal on the parent blob).

### 4. App (HARD — the one real blocker; do LAST or document if too costly)
`ManifestApp` needs a prebuilt `dist_files`/`bin_dist_files` payload because `standalone_v2` deploy requires compiled assets and there's no Vite in the test stack (`app_model` must be `"standalone_v2"`, deploy rejects others). The emitted transport extras (`repo_path`, `logo_b64`, `logo_content_type`, `src_files`, `bin_files`, `dist_files`, `bin_dist_files`) are already in `EXTRA_FIELD_POLICY`. Build a fixture that seeds a prebuilt dist (look at `test_solution_roundtrip.py` for any existing app-with-dist fixture to reuse). If this proves genuinely too costly for now, it's acceptable to leave App documented-as-uncovered with the precise blocker — but make that an EXPLICIT statement, not a silent gap.

### 5. FIX auto_fill (it's a REAL DROP, not a product decision — verdict corrected)
`FormField.auto_fill` is `dict[str, str]` mapping THIS-form sibling-field-names → data-provider metadata paths (e.g. `{"employee_count": "recommended_employee_count"}`), validated to reference fields that exist in the same form (`contracts/forms.py:93,143`). It is **self-contained portable form structure** (like `form_schema`), NOT an env-specific binding — so it SHOULD travel, and the detector correctly found a real drop. Fix BOTH halves:
- **Serialize:** `_form_field_to_schema_dict` (`manifest_generator.py:104-130`) must include `auto_fill` in the emitted field dict.
- **Parse:** the form indexer (`api/src/services/file_storage/indexers/form.py`) must read `auto_fill` back when reconstructing form fields.
- Then the existing `test_form_field_auto_fill_is_dropped_below_manifest` test (which currently PINS the drop) must be inverted to assert it now ROUND-TRIPS. Mirror the Phase-1 `tool_description` / the Phase-1.5 agent-limit fixes.
